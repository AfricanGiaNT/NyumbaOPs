# 🌍 PUBLIC UI/UX — LOCAL BOOKINGS & ENQUIRIES

---

## 🎯 Purpose of the Public Website

For **local users**, the website should:

1. Show available apartments clearly
2. Build trust (this is real, not a scam)
3. Make it easy to:
    - Request a booking, OR
    - Send an enquiry
4. Avoid back-and-forth on WhatsApp for basics

👉 **This is NOT an Airbnb replacement**

👉 This is a **lead + booking request machine**

---

## 🧠 Public UX Principles (Non-Negotiable)

1. **Mobile-first only**
2. **Few pages, very clear**
3. **Prices visible**
4. **No forced account creation**
5. **Booking request, not instant booking**
6. **Images do most of the selling**
7. **Fast on bad internet**
8. **Accessible to all users**
9. **Graceful error handling**
10. **Works on low-end devices**

---

## 🧭 Public Site Pages

1. Home
2. Properties List
3. Property Detail
4. Booking / Enquiry Form
5. Confirmation Page

That's it.

### Database Schema Addition

**New Table: `legal_documents`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | ✅ | Primary key |
| type | Enum | ✅ | TERMS_OF_SERVICE, PRIVACY_POLICY |
| version | String(20) | ✅ | e.g., "1.0", "1.1" |
| content | Text | ✅ | Full document content (markdown) |
| effective_date | Date | ✅ | When this version takes effect |
| is_current | Boolean | ✅ | Is this the active version |
| created_by | UUID | ✅ | FK → users |
| created_at | Timestamp | ✅ | |
```sql
CREATE TYPE legal_document_type AS ENUM ('TERMS_OF_SERVICE', 'PRIVACY_POLICY');

CREATE TABLE legal_documents (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  type legal_document_type NOT NULL,
  version VARCHAR(20) NOT NULL,
  content TEXT NOT NULL,
  effective_date DATE NOT NULL,
  is_current BOOLEAN NOT NULL DEFAULT false,
  created_by UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  
  UNIQUE (type, version)
);

CREATE INDEX legal_documents_current_idx ON legal_documents (type) WHERE is_current = true;
```

**New Table: `legal_acceptances`**

Track when guests accept terms:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | ✅ | Primary key |
| guest_id | UUID | | FK → guests (NULL for anonymous) |
| inquiry_id | UUID | | FK → inquiries |
| document_type | Enum | ✅ | Which document |
| document_version | String(20) | ✅ | Version accepted |
| ip_address | String(45) | | For audit |
| user_agent | String(500) | | For audit |
| accepted_at | Timestamp | ✅ | |
```sql
CREATE TABLE legal_acceptances (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  guest_id UUID REFERENCES guests(id) ON DELETE SET NULL,
  inquiry_id UUID REFERENCES inquiries(id) ON DELETE SET NULL,
  document_type legal_document_type NOT NULL,
  document_version VARCHAR(20) NOT NULL,
  ip_address VARCHAR(45),
  user_agent VARCHAR(500),
  accepted_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX legal_acceptances_guest_idx ON legal_acceptances (guest_id);
CREATE INDEX legal_acceptances_inquiry_idx ON legal_acceptances (inquiry_id);
```

### Critical UX Choices

- **Two CTAs only** (Request Booking, Ask a Question)
- No calendar complexity
- Images do the heavy lifting
- Verified badge builds trust
- Response time sets expectations
- Share functionality for word-of-mouth
- **Guest reviews build social proof** (curated testimonials)
- **FAQ link reduces repetitive inquiries**
- **Similar properties keep users browsing** (increases booking chances)

### Guest Reviews Section Rules

- Show maximum 2-3 reviews on detail page
- "View All" links to full reviews modal/page
- Only show section if property has 1+ published testimonials
- Display average rating and total count
- Reviews are curated by owner (not user-generated for MVP)

### Similar Properties Section Rules

- Show maximum 4 similar properties (2 visible, swipe for more)
- Similarity based on: same area, similar price (±30%), similar capacity
- Exclude current property and inactive properties
- Hide section entirely if fewer than 2 similar properties available
- Compact card design: image, name, price, view button only

### FAQ Link Rules

- Links to dedicated FAQ page
- Positioned after CTAs but before similar properties
- Helps reduce repetitive inquiry questions

# 🤔 BOOKING REQUEST vs ENQUIRY

---

## When to Show Booking Request Form

- User clicks "Request Booking" on property detail
- User knows their dates
- Intent: "I want to book this specific property for these dates"

---

## When to Show Enquiry Form

- User clicks "Ask a Question" on property detail
- User has questions before committing
- Intent: "I need more information before deciding"

---

## Key Differences

| Aspect | Booking Request | Enquiry |
| --- | --- | --- |
| Dates required | Yes | No |
| Guests required | Yes | No |
| Message | Optional | Required |
| Follow-up | Call to confirm booking | Answer question first |
| Urgency | Higher | Lower |

---

## In Database

- Both stored in `inquiries` table
- Differentiated by `type` field:
    - `BOOKING_REQUEST`
    - `ENQUIRY`
- Different status flows:
    - Booking: NEW → CONTACTED → CONVERTED → (booking created)
    - Enquiry: NEW → RESPONDED → CLOSED

| answer | Text | ✅ | | The answer (supports markdown) |
| display_order | Integer | ✅ | 0 | Sort within category |
| is_published | Boolean | ✅ | true | Display publicly |
| created_by | UUID | ✅ | | FK → users |
| created_at | Timestamp | ✅ | now() | |
| updated_at | Timestamp | ✅ | now() | |
```sql
CREATE TABLE faqs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  category VARCHAR(50) NOT NULL,
  question VARCHAR(255) NOT NULL,
  answer TEXT NOT NULL,
  display_order INTEGER NOT NULL DEFAULT 0,
  is_published BOOLEAN NOT NULL DEFAULT true,
  created_by UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX faqs_category_idx ON faqs (category, display_order) WHERE is_published = true;
```

# 🔐 PUBLIC FORM SECURITY

---

## Spam Prevention

- **Honeypot field:** Hidden field that bots fill (humans don't see it)
- **Rate limiting:** 5 submissions per IP per 15 minutes
- **Bot detection:** Submission in less than 3 seconds = suspicious
- **Phone validation:** Must be valid Malawi format

---

## Input Sanitization

- Strip HTML tags from all text inputs
- Validate phone number format server-side
- Limit message length: 500 characters max
- Escape special characters

---

## Data Privacy

- Don't store IP addresses long-term
- Phone numbers visible only to owner/staff
- No guest data shared with third parties
- No tracking cookies (privacy-friendly analytics only)

# 🧠 DESIGN PHILOSOPHY SUMMARY

---

1. **Mobile-first, always** – Most users are on phones
2. **Fast on bad internet** – Optimize everything
3. **Trust before conversion** – People need to believe it's real
4. **Human in the loop** – No instant booking, manual confirmation
5. **Reduce friction** – No accounts, minimal form fields
6. **Clear expectations** – Tell users exactly what happens next
7. **Graceful failures** – Every error has a recovery path
8. **Local context** – Phone > email, WhatsApp is king
